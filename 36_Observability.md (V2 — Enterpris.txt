36_Observability.md (V2 — Enterprise Fintech Observability System)
1. OBJECTIF DU SYSTÈME

Ce module définit la visibilité totale du système Amanah en production.

👉 Objectif :

voir tout ce qui se passe dans le système en temps réel
détecter anomalies instantanément
tracer chaque transaction de bout en bout
comprendre performance + erreurs + comportements utilisateurs
assurer contrôle bancaire-grade
2. CORE PRINCIPLE
2.1 Full visibility requirement
If it happens in the system, it must be observable.
2.2 Three pillars
Logs (what happened)
Metrics (how much / how fast)
Traces (end-to-end journey)
3. LOGGING SYSTEM
3.1 Structure
{
  "timestamp": "ISO-8601",
  "level": "info | warn | error",
  "service": "payment-service",
  "traceId": "uuid",
  "message": "Payment processed",
  "metadata": {}
}
3.2 Rules
structured logs only (JSON)
no plaintext logs in production
every log MUST include traceId
4. METRICS SYSTEM
4.1 Core metrics
transaction volume
payment success rate
ledger commit latency
FX conversion time
fraud detection rate
4.2 Performance metrics
API latency < 300ms
Payment processing < 2s
Ledger commit < 100ms
4.3 Infrastructure metrics
CPU usage
memory usage
pod scaling events
queue depth
5. TRACING SYSTEM
5.1 End-to-end trace model
User request
 → API Gateway
 → Payment Service
 → Risk Engine
 → FX Engine
 → Ledger Service
 → Response
5.2 Rule
Every request must have a full trace across all services.

7. ALERTING SYSTEM
7.1 Alert types
critical system failure
ledger imbalance detection
fraud spike detection
FX failure spike
7.2 Severity levels
LOW → info notification

HIGH → engineering alert

Every critical anomaly must trigger real-time alerting.
