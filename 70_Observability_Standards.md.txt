69_Observability_Standards.md
(V3 — Enterprise Observability, Monitoring & Operational Intelligence Framework)
1. OBJECTIF

Ce document définit les standards d'observabilité obligatoires pour tous les composants Amanah.

Il couvre :

logs ;
métriques ;
traces distribuées ;
monitoring infrastructure ;
monitoring métier ;
alerting ;
SLI/SLO/SLA ;
gestion des incidents ;
dashboards opérationnels.

L'objectif :

Détecter, comprendre et résoudre les problèmes avant qu'ils n'affectent les clients.

2. CORE PRINCIPLES
2.1 Observability by Default
Every service must be observable from day one.

Aucun service ne peut être déployé sans instrumentation.

2.2 Three Pillars of Observability

Amanah utilise trois piliers :

Logs

Metrics

Traces
2.3 Business + Technical Observability

Une banque ne surveille pas seulement les serveurs.

Elle surveille aussi :

paiements ;
transactions ;
soldes ;
risques ;
conformité ;
expérience client.

4. DISTRIBUTED TRACING

Tous les appels inter-services utilisent un :

Trace ID

Exemple :

Customer Request

↓

API Gateway

↓

Payment Service

↓

Compliance Engine

↓

Ledger

↓

Notification

Toutes les étapes partagent le même identifiant.

5. OPENTELEMETRY STANDARD

Amanah utilise :

OpenTelemetry ;
traces distribuées ;
métriques standardisées ;
instrumentation automatique.

Chaque service doit exporter :

traces ;
métriques ;
logs structurés.
6. LOGGING STANDARDS

Tous les logs sont structurés.

Format :

{
 "timestamp":"2026-01-01T10:00:00Z",
 "level":"INFO",
 "service":"payment-service",
 "tenantId":"uuid",
 "traceId":"uuid",
 "message":"Payment completed"
}
7. LOG LEVELS

Niveaux :

Niveau	Usage
ERROR	Erreur nécessitant action
WARN	Situation anormale
INFO	Événement normal
DEBUG	Développement uniquement
8. SECURITY LOGGING

Les logs ne doivent jamais contenir :

mot de passe ;
PIN ;
CVV ;
clé privée ;
données personnelles non nécessaires.

15. ALERTING RULES

Les alertes doivent être :

actionnables ;
documentées ;
prioritaires.

Exemple :

Mauvaise alerte :

CPU élevé

Bonne alerte :

Payment Service latency >500ms for 5 minutes affecting 20% transactions
16. INCIDENT SEVERITY

17. SLI / SLO / SLA
SLI

Mesure réelle.

Exemple :

Temps réponse API.

SLO

Objectif interne.

Exemple :

99.95% disponibilité.

SLA

Engagement client.

Exemple :

99.9% uptime contractuel.

25. CLAUDE CODE RULES

Claude Code must:

ALWAYS add observability to new services.
ALWAYS include structured logging.
ALWAYS propagate Trace IDs.
ALWAYS expose health metrics.
NEVER log sensitive information.
ALWAYS define meaningful alerts.
ALWAYS monitor business and technical metrics.
ALWAYS support distributed tracing.
ALWAYS document operational dashboards.
ALWAYS follow Enterprise Observability Standards.
26. SUCCESS CRITERIA

Le standard Observability est respecté lorsque :

chaque service est visible en production ;
les incidents sont détectés rapidement ;
les causes racines peuvent être identifiées ;
les performances métier et techniques sont surveillées ;
les opérations bancaires restent transparentes et contrôlées.
