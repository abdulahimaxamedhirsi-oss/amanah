63_API_Design_Standards.md
(V3 — Enterprise API Architecture & Governance Standards)
1. OBJECTIF

Ce document définit les standards obligatoires pour toutes les APIs Amanah.

Il garantit :

cohérence entre les services ;
compatibilité long terme ;
sécurité ;
facilité d'intégration ;
expérience développeur professionnelle.

Toutes les APIs internes et externes doivent respecter ces règles.

2. CORE PRINCIPLES
2.1 API First
Every capability must be designed as an API before implementation.

L'API est le contrat principal.

Le frontend, les applications mobiles, les partenaires et les services internes utilisent les mêmes contrats.

2.2 Backward Compatibility
Existing clients must continue working.

Une nouvelle version ne doit jamais casser les intégrations existantes sans migration planifiée.

2.3 Security by Default

Chaque API doit être sécurisée dès sa création.

5. URL CONVENTIONS

Format obligatoire :

https://api.amanah.com/v1/resource

Exemples :

Créer un client :

POST /v1/customers

Consulter un compte :

GET /v1/accounts/{accountId}

Créer un paiement :

POST /v1/payments
6. VERSIONING

Toutes les APIs publiques utilisent un versionnement.

Exemple :

/api/v1/


Nouvelle évolution majeure :

/api/v2/

12. IDEMPOTENCY

Toutes les opérations financières doivent supporter l'idempotence.

Obligatoire pour :

paiements
transferts
remboursements
virements
création de transactions

Exemple :

Header :

Idempotency-Key: uuid

Une même requête ne doit jamais créer deux transactions.

13. PAGINATION

Exemple :

"total":500
}

25. DEPRECATION POLICY

Une API abandonnée doit :

être annoncée ;
avoir une date de fin ;
proposer une migration ;
rester disponible pendant une période définie.
26. CLAUDE CODE RULES

Claude Code must:

ALWAYS design APIs API-First.
NEVER break backward compatibility.
ALWAYS version public APIs.
ALWAYS implement idempotency for financial operations.
ALWAYS use standardized error responses.
NEVER expose internal database models directly.
ALWAYS document APIs using OpenAPI.
ALWAYS secure APIs through the API Gateway.
ALWAYS implement rate limiting.
ALWAYS maintain API governance metadata.
27. SUCCESS CRITERIA

Le standard API est respecté lorsque :

toutes les APIs suivent une convention commune ;
les intégrations partenaires sont simples et stables ;
les opérations financières sont protégées contre les doublons ;
les API sont documentées et versionnées ;
les changements peuvent évoluer sans casser les clients existants.
