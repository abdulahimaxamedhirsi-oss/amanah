64_Database_Design_Standards.md
(V3 — Enterprise Database Architecture & Governance Standards)
1. OBJECTIF

Ce document définit les standards obligatoires pour la conception, l'utilisation et la maintenance des bases de données Amanah.

Il garantit :

intégrité financière ;
haute disponibilité ;
performance ;
sécurité ;
auditabilité ;
évolutivité.
2. CORE PRINCIPLES
2.1 Data Integrity First
Financial data must always remain accurate and consistent.

Aucune optimisation ne doit compromettre l'intégrité des données.

2.2 Database Per Domain
Each business domain owns its data.

Un service ne doit jamais accéder directement à la base d'un autre domaine.

2.3 Immutable Financial Records
Financial history must never be modified or deleted.

Les transactions financières sont immuables.

6. PRIMARY KEYS

Standard :

UUID obligatoire.

Exemple :

7f8a92d4-xxxx-xxxx

Pourquoi :

distribution mondiale ;
multi-tenant ;
sécurité ;
réplication.
7. TENANT ISOLATION

Toutes les tables multi-tenant contiennent :

tenant_id

Exemple :

accounts

id

tenant_id

customer_id

balance
8. FINANCIAL PRECISION

Jamais utiliser :

FLOAT
DOUBLE

Pour l'argent.

Utiliser :

DECIMAL
NUMERIC

Exemple :

amount DECIMAL(18,2)
9. TRANSACTION MANAGEMENT

Les opérations financières utilisent :

ACID transactions ;
isolation appropriée ;
verrouillage contrôlé ;
idempotency keys.
10. DATABASE MIGRATIONS

Toutes les modifications passent par :

migrations versionnées ;
scripts automatisés ;
validation CI/CD.

Interdit :

Modification manuelle en production.

11. INDEXING STANDARDS

Chaque table critique possède :

index primaire ;
index recherche ;
index tenant ;
index temporel.

Exemple :

INDEX(customer_id)

INDEX(created_at)

INDEX(tenant_id)

19. SOFT DELETE

Les données métier importantes ne sont pas supprimées.

Utiliser :

deleted_at

ou :

status = CLOSED
20. LEDGER DATABASE RULES

Le Ledger doit respecter :

double entrée ;
append-only ;
balance dérivée des écritures ;
aucune modification historique ;
correction par nouvelle écriture.

Exemple :

Erreur :

Transaction incorrecte

Correction :

Reversal Transaction

Jamais :

UPDATE transaction

25. CLAUDE CODE RULES

Claude Code must:

NEVER directly access another domain database.
ALWAYS respect database ownership.
NEVER store financial amounts using floating types.
ALWAYS use migrations.
ALWAYS include tenant isolation where required.
NEVER modify immutable financial records.
ALWAYS create proper indexes.
ALWAYS consider scalability before schema design.
ALWAYS encrypt sensitive database fields.
ALWAYS follow Enterprise Database Standards.
26. SUCCESS CRITERIA

Le standard Database est respecté lorsque :

les données financières restent exactes et immuables ;
chaque domaine possède son propre contrôle des données ;
les bases peuvent évoluer à grande échelle ;
les performances restent constantes ;
les audits et restaurations sont possibles à tout moment.
