1. OBJECTIF DU SYSTÈME

Le Global Multi-Tenant Architecture permet à Amanah d'héberger plusieurs organisations indépendantes sur une infrastructure unique.

Chaque tenant fonctionne comme s'il disposait de sa propre plateforme bancaire.

Les données, configurations, utilisateurs, politiques, clés de chiffrement et ressources sont totalement isolés.

2. CORE PRINCIPLES
2.1 Complete Tenant Isolation
Each tenant must operate as an independent financial institution.
2.2 Shared Infrastructure
Infrastructure is shared.
Data is never shared.
2.3 Configuration over Customization
Tenant differences are configuration-driven, never hardcoded.

22. CLAUDE CODE RULES

Claude Code must:

ALWAYS isolate tenant data.
NEVER allow cross-tenant access.
ALWAYS filter every query by tenantId.
NEVER hardcode tenant configuration.
ALWAYS support white-label branding.
ALWAYS isolate encryption keys.
ALWAYS support country-specific regulations.
NEVER mix audit logs between tenants.
ALWAYS enforce tenant quotas.
ALWAYS design services to scale horizontally across thousands of tenants.
23. SUCCESS CRITERIA

Le Global Multi-Tenant Architecture est considéré comme complet lorsque :

plusieurs institutions financières peuvent partager la même plateforme sans partage de données ;
chaque tenant est entièrement configurable et personnalisable ;
l'isolation des données, des politiques et des clés est garantie ;
la plateforme supporte plusieurs pays, devises et langues ;
le provisionnement d'un nouveau tenant est automatisé et rapide.
24. FUTURE EXTENSIONS (V4 READY)

L'architecture est conçue pour intégrer :

Hierarchical Tenants (groupe bancaire → filiale → agence).
Tenant Migration Engine pour déplacer un tenant entre régions ou infrastructures sans interruption de service.
