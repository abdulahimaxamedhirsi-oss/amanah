49_Plugin_SDK.md (V3 — Enterprise Plugin & Extension System)
1. OBJECTIF DU SYSTÈME

Le Plugin SDK permet d’étendre Amanah sans modifier le cœur du système.

Il autorise des modules externes à :

ajouter des fonctionnalités
écouter des événements
enrichir les workflows
créer des APIs additionnelles
personnaliser les règles métier
intégrer des systèmes externes

Tout en respectant une isolation stricte du Core Banking.

2. CORE PRINCIPLE
2.1 Core is Immutable
The core banking system must never be modified by plugins.

Les plugins peuvent :

✔ s’abonner
✔ enrichir
✔ observer
✔ proposer

Les plugins ne peuvent jamais :

✖ modifier le Ledger
✖ modifier les transactions
✖ bypass compliance
✖ accéder aux clés système

2.2 Event-Driven Extensibility
Plugins interact only through events and APIs.

9. PERMISSIONS SYSTEM

Plugins peuvent demander :

read-only access to events
access to anonymized data
API access
analytics access

Mais jamais :

accès direct Ledger
accès secrets
accès clés privées
10. SECURITY MODEL
10.1 Isolation

Chaque plugin est isolé :

process-level isolation
container-level isolation
network policies
token-based access
10.2 Signature Requirement

Tous les plugins doivent être :

signés
vérifiés
validés avant déploiement
11. RATE LIMITING

Chaque plugin a :

quota CPU
quota mémoire
quota API calls
quota events
12. ERROR HANDLING

Un plugin défaillant :

est isolé
ne bloque pas le core
est redémarré automatiquement
peut être désactivé
13. PLUGIN MARKETPLACE

Amanah peut héberger un marketplace :

plugins internes
plugins partenaires
plugins third-party

Chaque plugin est soumis à validation sécurité.

14. VERSIONING SYSTEM

semantic versioning

rollback automatique
15. OBSERVABILITY

16. PERFORMANCE OBJECTIVES

Event dispatch :

< 10 ms

Plugin execution :

< 100 ms (soft limit)

Sandbox startup :

< 200 ms

17. CLAUDE CODE RULES

Claude Code must:

NEVER allow plugins to modify Ledger or Treasury.
ALWAYS enforce sandbox isolation for plugins.
NEVER allow plugins to access secrets or private keys.
ALWAYS validate plugin signatures before activation.
NEVER allow plugins to bypass event bus.
ALWAYS enforce rate limits per plugin.
NEVER let a plugin failure impact core banking.
ALWAYS require explicit permissions for data access.
ALWAYS log all plugin executions with traceId.
ALWAYS support plugin rollback and versioning.
18. SUCCESS CRITERIA

Le Plugin SDK est considéré comme complet lorsque :

des fonctionnalités peuvent être ajoutées sans toucher au core banking ;
les plugins sont totalement isolés du Ledger et des systèmes financiers critiques ;
un système de sandbox sécurisé empêche toute escalade de privilèges ;
les événements sont distribués de manière fiable et scalable ;
un marketplace de plugins peut fonctionner en production ;
le système supporte des centaines voire milliers de plugins simultanément sans impact sur le cœur du système.
