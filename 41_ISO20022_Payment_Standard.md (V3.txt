41_ISO20022_Payment_Standard.md (V3 — Enterprise Financial Messaging Architecture)
1. OBJECTIF DU SYSTÈME

Ce document définit l'intégration complète du standard ISO 20022 dans Amanah.

ISO 20022 est le standard mondial utilisé par :

banques centrales
banques commerciales
SWIFT
SEPA
FedNow
TARGET
instant payments
infrastructures de règlement

L'objectif est que tous les messages financiers produits par Amanah soient compatibles avec ISO 20022, même si certains partenaires utilisent encore d'autres formats.

2. CORE PRINCIPLE
2.1 ISO 20022 is the canonical language
Every financial message inside Amanah must be representable as an ISO 20022 message.

Autrement dit :

Le modèle interne d'Amanah est plus riche que les API publiques.

Toutes les API REST devront pouvoir être converties vers ISO 20022.

2.2 Internal Canonical Model

Toutes les données financières passent par un modèle interne unique.

REST API
      │
      ▼
Canonical Financial Model
      │
      ▼
ISO 20022
      │
      ▼
External Networks

Aucun composant ne convertit directement REST → SWIFT.

La conversion passe toujours par le modèle canonique.

8. VERSION MANAGEMENT

ISO évolue.

Le système doit supporter plusieurs versions simultanément.

Exemple :

ISO20022 2019

ISO20022 2022

ISO20022 2025

Une banque partenaire peut utiliser une ancienne version.

Le moteur doit convertir automatiquement.

16. CLAUDE CODE RULES

Claude Code must:

NEVER couple business logic to ISO XML structures.
ALWAYS use the Canonical Financial Model as the intermediary.
NEVER process invalid ISO messages.
ALWAYS validate mandatory fields before transformation.
ALWAYS support multiple ISO versions simultaneously.
NEVER lose the original source message.
ALWAYS archive both canonical and original representations.
ALWAYS generate deterministic message identifiers.
NEVER bypass schema validation.
ALWAYS preserve end-to-end traceability.
17. SUCCESS CRITERIA

Le système est considéré comme conforme lorsque :

tous les paiements internes peuvent être représentés en ISO 20022 ;
plusieurs versions du standard peuvent coexister ;
les transformations sont réversibles et auditables ;
les messages sont validés, signés, archivés et traçables ;
le Ledger reste indépendant du protocole de messagerie.
