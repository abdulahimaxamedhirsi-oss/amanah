1. OBJECTIF DU SYSTÈME

Le système de Disaster Recovery (DR) garantit qu'Amanah puisse continuer à fonctionner ou être restauré rapidement après une catastrophe, sans compromettre l'intégrité financière.

Une catastrophe peut être :

panne totale d'un datacenter
corruption de base de données
cyberattaque
panne cloud
erreur humaine
catastrophe naturelle
perte réseau majeure
compromission d'une région entière

L'objectif n'est pas seulement de restaurer les serveurs.

L'objectif est de préserver l'intégrité financière absolue.

2. CORE PRINCIPLE
2.1 Money First
Financial integrity always has higher priority than service availability.

Si un choix doit être fait entre :

continuer le service avec un risque de corruption

ou

arrêter temporairement les paiements

Amanah choisit toujours :

arrêter les paiements.

2.2 No inconsistent recovery
The system must never recover into an inconsistent financial state.
3. BUSINESS CONTINUITY OBJECTIVES

Temps maximal acceptable avant reprise.

Objectifs :

≤ 10 minutes

Paiements

≤ 15 minutes

Analytics

Objectif Amanah :

Paiements :

0 seconde

Wallet :

Logs :

≤ 1 minute

Analytics :

≤ 15 minutes

4. DISASTER LEVEL CLASSIFICATION
Niveau 1 — Incident Mineur

Exemples

pod Kubernetes crash
API lente
worker arrêté

Action

Auto-redémarrage.

Aucun impact utilisateur.

Niveau 2 — Incident Moyen

Exemples

service Payment indisponible
Redis indisponible
queue saturée

Action

Failover local.

Niveau 3 — Incident Majeur

Exemples

PostgreSQL primaire perdu
corruption partielle
cluster entier indisponible

Action

Basculer vers région secondaire.

Niveau 4 — Catastrophe Totale

Exemples

région cloud entière détruite
cyberattaque globale
perte du datacenter principal

Action

Activation complète du Disaster Recovery Plan.

10. WALLET RECONSTRUCTION

Les wallets ne sont jamais restaurés directement.

Ils sont reconstruits.

Ledger

↓

Replay

↓

Balance calculation

↓

Wallet regenerated

Ainsi :

Aucune corruption de solde.

↓

Crash

18. CLAUDE CODE RULES

Claude Code must:

NEVER restore Wallet directly from backups.
ALWAYS rebuild Wallet from Ledger.
NEVER start Payment Engine before Ledger validation.
ALWAYS verify hashes after recovery.
NEVER ignore replication status.
ALWAYS replay events in chronological order.
NEVER allow recovery with reconciliation errors.
ALWAYS stop recovery if Ledger integrity cannot be proven.
NEVER prioritize uptime over financial integrity.
ALWAYS execute post-recovery audit before reopening production.
19. SUCCESS CRITERIA

Le Disaster Recovery est considéré comme réussi uniquement si :

aucune perte financière
RPO du Ledger = 0
RTO respecté
aucune transaction dupliquée
aucune transaction perdue
Ledger 100 % cohérent
Wallets reconstruits correctement
Hash cryptographiques identiques
Audit post-restauration validé
système totalement opérationnel
